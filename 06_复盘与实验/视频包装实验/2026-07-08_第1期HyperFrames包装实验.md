## 已确认方向

- 昨天用 `ffmpeg` 直接生成视频，效果很差；后续不把“纯 ffmpeg 合成”作为视频包装主路线。
- `ffmpeg/ffprobe` 后续只承担必要基础角色：抽音频、转码、补关键帧、读取媒体信息、最终音视频流校验。
- 今天优先尝试 HyperFrames：用 HTML/CSS/GSAP 做包装卡片、新闻画面、字幕和动效，再渲染成视频。
- 对具体新闻的视觉包装要尽量“对应正在说的内容”，不能只铺无关背景图。

## 素材策略

### 优先级 1：真实来源素材

适用内容：

- 官方文档截图。
- GitHub 项目页面截图。
- 官方发布页截图。
- 新闻源页面截图。
- 项目 README、release note、star 数、功能列表截图。

优点：真实、可追溯、和新闻强相关。  
风险：页面版权和平台使用边界要谨慎；不要截取与内容无关或误导性的局部。

### 优先级 2：强关联视频素材

适用内容：

- 新闻主体人物出镜视频。
- 官方发布会、演讲、播客、产品 demo。
- 开源项目官方演示视频。

优点：画面更有动感。  
风险：来源、授权、剪辑引用边界更复杂；如果不能确认来源和可用性，不应硬用。

### 优先级 3：AI 生图补充

适用内容：

- 抽象概念：AI 办公、智能体处理文件、多语言语音识别。
- 没有合适公开视觉素材的过渡画面。
- 统一风格的封面或章节卡。

工具：`right-codes-draw`，默认模型 `gpt-image-2`。  
原则：生图只能补充抽象表达，不能伪造成新闻现场、真实产品截图、真实人物事件图。

## 你和 Codex 的分工边界

### Codex 可以代劳

- 按录制文本拆分时间轴和包装段落。
- 根据字幕稿核对 ASR 初稿，纠正英文产品名和术语。
- 搜索、打开、截图可公开访问的网页资料，并记录来源链接。
- 设计 HyperFrames storyboard、图文卡、动效、字幕样式。
- 调用 `right-codes-draw` 生成抽象补充图，并报告真实输出路径和失败原因。
- 在依赖齐全后跑 `lint -> validate -> inspect -> snapshot/preview -> render`。
- 渲染后用 `ffprobe` 校验最终视频是否有视频流和音频流。
- 把每次实验结果、阻塞点和可复用流程写回 Markdown。

### 用户更适合完成

- 决定最终发布平台和画幅：横屏、竖屏或一版多裁切。
- 决定真人画面保留比例：全程出镜、半屏出镜、还是重点段落出镜。
- 判断哪些新闻素材可以接受截图引用，哪些需要更保守。
- 对最终口播语气、事实表达、观点强度做把关。
- 判断第 1 期是否需要加关注、社群或个人介绍钩子。

### 必须先问或明确阻塞的事项

- 需要下载、搬运、剪入第三方视频原片时，先确认来源和用途。
- 需要使用付费素材、登录账号、绕过访问限制时，先问用户。
- 来源不确定、页面不可访问、数据可能过期时，不编造、不补细节。
- 事实核验没有完成时，包装稿只能标记为“待核验”，不能当作可发布定稿。

## 建议的下一步

1. 先安装或定位 `ffmpeg/ffprobe`，解除 HyperFrames 渲染硬阻塞。
2. 用粗剪视频抽音频或直接导入，生成/整理字幕时间轴。
3. 按录制文本拆成 7-9 个包装段落，先做一个 20-30 秒小样。
4. 小样优先覆盖 2 个场景：真实网页截图卡片 + 一个抽象 AI 工作流动效。
5. 用户看小样后，再决定是否全片包装、是否生图、是否找强关联视频素材。

## 当前判断

HyperFrames 值得试，但不要预设它一定能一次达到发布级。最负责任的推进方式是：先跑 20-30 秒真实小样，把字幕、截图、动效、口播画面共存的问题暴露出来，再决定是否扩展到整条视频。

## 2026-07-08 测试版 v1 结果

测试目录：`06_复盘与实验/视频包装实验/2026-07-08_hyperframes_test_v1/`

本轮实际做了一个 HyperFrames 包装测试版：

- `public/index.html`：真人口播视频 + 右侧 AI 新闻卡片 + 底部字幕条 + 关键词 ticker。
- `preview-browser-5s-final.png`：浏览器真实预览截图。
- `transcoded/input-video-12s-h264.m4v`：为验证浏览器播放临时转出的 12 秒 H.264 小段。

验证结果：

| 项目 | 结果 | 说明 |
| --- | --- | --- |
| HyperFrames lint | 通过 | 0 error，0 warning |
| HyperFrames validate | 通过 | 0 runtime error，0 warning，0 contrast failure |
| HyperFrames inspect | 通过 | 0 layout error，0 warning |
| Chrome 真实预览 | 通过 | 视频可播放，包装层显示正常 |
| HyperFrames render MP4 | 通过 | 已生成 12 秒测试 MP4：`2026-07-08_hyperframes_test_v1/test-output.mp4` |

关键结论：

- “全部用 HyperFrames 做包装层”是可行的，至少浏览器预览层面已经跑通。
- 不能把最终 MP4 导出也理解成完全不需要编码器；HyperFrames 渲染 MP4 仍需要底层 `ffmpeg/ffprobe`。
- 原始粗剪视频是 HEVC + AAC、1080x1920，后续为了稳定预览和渲染，最好先导出为 H.264 MP4。
- 这一步不等于回到昨天的纯 `ffmpeg` 包装方案；正确分工是：HyperFrames 做画面设计和时间线，编码器只负责最终视频格式输出。

测试 MP4 校验：

- 路径：`06_复盘与实验/视频包装实验/2026-07-08_hyperframes_test_v1/test-output.mp4`
- 文件大小：约 4.9MB。
- 时长：约 12.03 秒。
- 视频流：H.264，1920x1080，30fps。
- 音频流：AAC。
