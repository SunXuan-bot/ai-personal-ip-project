# 2026-07-08 竖屏完整 v1 视觉包装优化审查

记录时间：2026-07-08  
视频路径：`06_复盘与实验/视频包装实验/2026-07-08_vertical_full_v1/output-vertical-full-v1.mp4`  
审查方式：第一性原理 + 对抗式审查  

## 当前结论

这版已经验证了“真人口播视频 + HyperFrames 包装层 + 最终 MP4 渲染”这条技术链路可行，但还没有进入真正的视频包装阶段。当前效果更像“文本信息层叠加”，不是“新闻辅助视觉包装”。

核心问题不是 HyperFrames 做不到，而是当前缺少三个生产要素：

1. 真实时间轴：每条新闻准确从哪一秒开始。
2. 真实素材：每条新闻对应的网页截图、产品截图、图表或视频素材。
3. 包装规则：标题卡、说明卡、截图卡、数据卡分别什么时候出现、占多大、怎么动。

## 用户反馈对应修改

### 1. 底部黄色关键词栏

保留，不删除。

建议改法：

- 继续保留底部黄色 rail，作为栏目识别和关键词提示。
- 英文关键词保留，但不要抢字幕和主体视觉。
- 后续可以让当前讲到的关键词轻微高亮，例如 DeepSeek 段落只高亮 `DeepSeek`，其余词降低透明度。

### 2. 字幕

字幕不建议继续由 HyperFrames 自动生成或粗对齐。

更合理分工：

- 剪映负责识别、跟踪、校对、字幕样式。
- HyperFrames 负责标题卡、来源截图、数据卡、动效、画面节奏。
- 如果用户导出剪映字幕文件，优先使用 `.srt` 作为包装时间轴依据，而不是只用于显示字幕。

关键点：字幕文件不仅是字幕，也是最好的“口播真实时间轴”来源。

### 3. 左上角名称

保留“武汉AI小孙”。

建议保持小而稳定，不做强动效。它是账号识别，不是每条新闻的信息重点。

### 4. 右上角 AI NEWS 01

改为日期，例如：

```text
2026 年 7 月 8 日
```

如果担心太长，可以用：

```text
2026.07.08
```

竖屏里建议优先用 `2026.07.08`，信息清楚，视觉负担更轻。

### 5. 上方新闻标题色块

当前色块可以保留，但要从“固定大卡片”改成“内容自适应卡片”。

#### 5.1 标题开始帧不准

问题原因：

当前 `public/index.html` 里卡片时间是手写数组，例如：

```js
["#card-deepcode", 7.8, 23.2]
["#card-office", 23.6, 40.0]
```

这不是从真实字幕或音频分段来的，所以必然会漂。

建议解决方案：

1. 在剪映生成并校对字幕。
2. 导出 `.srt`。
3. 用每条新闻第一句的字幕开始时间，生成 `news_segments.json`。
4. HyperFrames 只读取 `news_segments.json`，不再手写时间。

示例结构：

```json
[
  {
    "id": "deepcode",
    "start": 7.42,
    "end": 23.10,
    "title": "01 DeepSeek 文档收录 Deep Code",
    "note": "支持深度思考、推理强度控制和 Agent Skills",
    "source": "DeepSeek API Docs + GitHub",
    "visual": "screenshots/deepseek-deepcode.png"
  }
]
```

这样“当条新闻出现的第一秒标题就出现”才有可靠依据。

#### 5.2 标题出现动效

建议：

- 背板不动或只做 0.12 秒轻微淡入。
- 文字做打印机、闪出、逐词出现三选一。
- 新闻标题用“短促出现”，补充说明用“延迟 0.3-0.5 秒淡入”。

更适合本账号的风格是“逐词闪出”，不要太像科技发布会，也不要花哨。

#### 5.3 色块大小

当前最大问题是卡片固定 `left:42px; right:42px; min-height:330px; top:144px;`，导致文字少时背板仍然很大，遮挡头发。

建议：

- 卡片从全宽大卡改为 `inline-fit` 宽度。
- 最大宽度控制在画面宽度 78%-84%。
- 高度由内容决定，不设置大 `min-height`。
- 默认放在上方安全区，如果人物头顶被挡，优先改成左上/右上短卡，而不是继续压人物。

#### 5.4 文案功能约束

标题卡只做一件事：告诉观众这条新闻是什么。

推荐规则：

- 标题功能：只保留 `序号 + 新闻标题`。
- 补充说明功能：只保留一个关键数据、一个来源或一个影响点。
- 不要把标题、解释、来源、数据全部塞进同一张卡。

## 推荐提效流程

### 用户负责

1. 在剪映里生成字幕并校对。
2. 导出 `.srt` 字幕。
3. 确认哪些第三方截图/视频可以接受用于发布。

### Codex 负责

1. 根据 `.srt` 生成 `news_segments.json`。
2. 采集公开网页截图，并记录来源。
3. 生成素材台账 `visual_assets.md`。
4. 生成 HyperFrames 包装页。
5. 渲染测试版，并输出抽帧 contact sheet。
6. 根据抽帧检查遮挡、溢出、时序漂移和视觉重复。

## 对抗式审查

### 可能错的地方

- 过度包装会遮挡人物，削弱真实感。
- 每条新闻都塞网页截图，会变成信息噪音，观众看不清。
- 视觉素材越多，越容易拖慢日更节奏。
- 如果每天都手工找素材、截图、裁切，账号会被制作流程拖死。
- 对新闻源截图使用不谨慎，可能带来版权或平台规则风险。

### 反向原则

- 不是每条新闻都必须配复杂素材。
- 每期只选 1-2 条做重点视觉，其余用轻量标题卡。
- 真正值得视觉化的是“可信来源”和“关键数据”，不是每一句话。
- 日更阶段优先稳定生产，不追求每条都像栏目片头。

## 下一版最小可行优化

建议 v2 不要一次性大改成复杂新闻视频。先做以下 6 个优化：

1. 去掉 HyperFrames 自带字幕层，字幕交给剪映。
2. 右上角改日期。
3. 标题卡改成自适应短卡，减少遮挡头发。
4. 标题文字增加逐词闪出动效。
5. 用剪映导出的 `.srt` 生成真实新闻分段时间。
6. 只给 Deep Code、OfficeCLI、Fun-ASR 三条接入真实网页截图，先验证“素材 + 真人 + 字幕”是否协调。

如果这 6 项跑通，再考虑加入视频素材或更复杂图表。
